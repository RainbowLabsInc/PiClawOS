## 2025-03-05 - Precompiled Regex for Multiple Substring Checks
**Learning:** For optimal performance on resource-constrained devices like the Raspberry Pi, pre-compiled regex searches (`re.compile`) are approximately 2x faster than generator expressions using `any()` with list or tuple iteration when checking if any of a set of keywords exists in a string. This is especially true when there's no match (which is the most common path) or when the target text is long.
**Action:** When performing substring matching against a set of multiple keywords (e.g. `any(kw in text for kw in keywords)`), replace it with a module-level pre-compiled regex. Combine the array of keywords into a single optimized regex (e.g. `(?:word1|word2)`), making sure to sort the keywords by length descending so the engine doesn't eagerly consume partial matches.

## 2024-03-24 - Pre-compiled Regex matching with Compound Words
**Learning:** For optimal performance when performing substring checks, using pre-compiled regex searches (`re.compile`) at the module level is significantly faster than using generator expressions with `any()`. However, when dealing with languages that rely heavily on compound words (like German, e.g., "Deckenlicht"), using strict word boundaries (`\b`) in the regex breaks partial matching. Additionally, when using the `|` OR operator without boundaries, it's essential to sort keywords by length descending so that longer substrings are matched first, preventing premature partial matches.
**Action:** When migrating intent keyword detection to `re.compile`, omit `\b` for languages that use compound words, and always sort the keywords with `sorted(..., key=len, reverse=True)` before joining them into the regex.
