## 2025-03-05 - Precompiled Regex for Multiple Substring Checks
**Learning:** For optimal performance on resource-constrained devices like the Raspberry Pi, pre-compiled regex searches (`re.compile`) are approximately 2x faster than generator expressions using `any()` with list or tuple iteration when checking if any of a set of keywords exists in a string. This is especially true when there's no match (which is the most common path) or when the target text is long.
**Action:** When performing substring matching against a set of multiple keywords (e.g. `any(kw in text for kw in keywords)`), replace it with a module-level pre-compiled regex. Combine the array of keywords into a single optimized regex (e.g. `(?:word1|word2)`), making sure to sort the keywords by length descending so the engine doesn't eagerly consume partial matches.
## 2026-03-31 - Fast String Containment
**Learning:** Found that Python's native string containment (`in`) operator is significantly faster than pre-compiled regexes (`re.search`) or generator expressions (`any(...)`) for simple substring checks, especially when the target text is already lowercased. Additionally, redundant substring checks (e.g., checking for both "kleinanzeigen" and "kleinanzeigen.de") are unnecessary since the base keyword will inherently match the domain suffix.
**Action:** When performing simple substring matching against a known set of single keywords where exact word boundaries are not required, rely on the fast native `in` operator (e.g., `if "keyword" in text:`). Ensure that checks are logically simplified to avoid redundant matching overhead. Do not default to `re.IGNORECASE` if the input text is already transformed to lowercase earlier in the pipeline.
