## 2025-03-05 - Precompiled Regex for Multiple Substring Checks
**Learning:** For optimal performance on resource-constrained devices like the Raspberry Pi, pre-compiled regex searches (`re.compile`) are approximately 2x faster than generator expressions using `any()` with list or tuple iteration when checking if any of a set of keywords exists in a string. This is especially true when there's no match (which is the most common path) or when the target text is long.
**Action:** When performing substring matching against a set of multiple keywords (e.g. `any(kw in text for kw in keywords)`), replace it with a module-level pre-compiled regex. Combine the array of keywords into a single optimized regex (e.g. `(?:word1|word2)`), making sure to sort the keywords by length descending so the engine doesn't eagerly consume partial matches.
## 2024-05-19 - Fast String/Word Search lookups with Frozensets
**Learning:** Found an intent parsing loop (`_ha_shortcut` in `piclaw/agent.py`) doing inline list creations and concatenations inside a loop list comprehension (creating a new list for every single word checked), resulting in ~0.90s per 100k executions. While doing so, the code was accidentally triggering an exception on new setups: `TypeError: can only concatenate tuple (not "list") to tuple`.
**Action:** Lift array constructions like `["schalte", "mach"]` into module-level variables wrapped in `frozenset()`. This fixes the bug and reduces loop logic to single `not in` lookups, dropping runtime from ~0.90s to ~0.33s per 100k invocations without any sacrifice to readability. Also learned `E402` rules for top-level imports mean module constants must be placed AFTER all standard library module imports.
